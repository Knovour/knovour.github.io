React 一個月的開發感想

2016-05-21
react.js

簡介

React 是以函數式編程為原則來做開發，函數式編程幾個簡單的重點如：

  • 數值不可變：在一些比較嚴格規定的語言如 Haskell 的每個變數都是強制不可變的，而大部分的語言都是折衷分為可變與不可變。
  • 一個函數只做一件事：這表示函數要單純，所以要把處理的事情分的很清楚，對不是從函數式語言入門的人來說，要分多清楚端看過往寫程式的經驗以及潔癖程度。
  • 每次回傳都是一筆新的資料：基於數值不可變的特性。
  • 一樣的參數 => 一樣的結果：這點也包括不該改變參數原本的值，比如 slice 與 splice 這兩個函數都能夠回傳一個陣列指定範圍的切片，但 splice 不屬於純函數。

這會帶來什麼好處？

  • 耦合度低：每個函數都很單純，程式碼簡短（通常）。
  • 容易測試：沒有副作用，傳入與傳出的結果是可預測的。
  • 效率：有一個叫 Om 的項目，是以 ClojureScript 來寫 React 再轉成純 js，轉換後發現執行效率比用純 js 編寫高出了三倍，其中數值不可變是比較大的主因，促使 React 官方寫了 Immutable.js 以達成一樣的結果。

    結構

React 專案一般會分成兩個資料夾：

  • components：存放網頁單一元素的地方，比如按鈕、表單、卡片等會重複使用到的東西，程式碼會儘量保持簡單乾淨。
  • containers：載入各個 component 組合出一個頁面、進行數據傳遞以及一些事件綁定的函式建立等等，簡單講就是骯髒事儘量先在這邊處理再傳遞到各個 component。

建立一個 component 有三種方法：

  • 單一 function：最單純的方式，不需要做額外處理，只是單純回傳一個 component 時建議使用。
  • createClass：ES5 時的作法，官方範例也是以這種寫法為主，但在 ES6 的 class 成為主流後不再建議使用。
  • ES6 class：現在主流作法，但 ES6 的 class 實現不算完全，寫起來沒有 createClass 那樣有一體性的感覺。

觀念落差

以前的共識是 html、css、js 各自獨立不混雜，而 React 的事件綁定就是綁在 DOM 上（正確來講是 Virtual DOM），這點我其實還能接受；但鼓吹連 css 都要塞進去，卻讓我很難接受，尤其在看到像 Material-UI 這種一打開 console 就看到一大票的 inline css 實在很吐血。

但這也不是沒有好處，以往載入如 Bootstrap 等的 framework 時，如果不考慮直接修改原始碼來改變樣式時，很常需要另外寫 css 來覆蓋原本的設計，必要時還得補上 !important，如果用上如 [emantic U] 這種小元件很多又彼此有關聯的，那就有如一場災難。

jsx 裡的 css 以 jsON 格式儲存，如果想改變或新增部份的 Material-UI 裡的 css，只要寫進參數就可以直接替換。

程式碼難看

在 js 與 jsx 交雜的情況下，看程式碼是一件頗難過的事，怎麼寫麼排都覺得很難看。一部份原因是 ES6 的 class 只做了半套，如果想讓 class 的程式碼看起來更有一體性，引入審核中的 ES7 語法是個選擇。事實上，現在很多 React 相關的套件，很多在 example 都用上了 ES7 的語法糖，如果自己在開發時沒有用 ES7，那麼在看這些範例時，思維上常要做一點切換。
